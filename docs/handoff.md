# Session 175 handoff (2026-04-30) — Zoom DOM inventory shipped end-to-end

This session walked the DevTools-console probe-state-list workflow that was queued from the session-174 handoff. User opened a Zoom meeting in their daily Chrome (signed-in operator account); I sent self-contained IIFEs at each lifecycle state; user pasted output back; selectors landed verbatim in the new `docs/dom-inventory-zoom.md`. All architecturally-critical states are now logged: state 1 (launcher page), 2a (camera + mic permission modals — two sequential ReactModal dialogs; "Continue without microphone and camera" is a `<span role="button">`, not a `<button>`), 2b (preview + name input — `#input-for-name` is the only stable hook; Join enabled via `button.preview-join-button:not(.zm-btn--disabled)`), 4a (in-meeting toolbar), 4b (chat panel — chat input is `div.tiptap.ProseMirror[contenteditable]`, NOT a textarea, so Meet-style `page.fill()` won't work; per-message DOM has `data-userid` + `data-name` on `.chat-item__sender` and a UUID `data-id` on the row for dedup), 4c (More menu), 4c.1–4c.3 (Captions submenu + first-run language modal + caption rendering), 4d (participants count + roster — count exposed in toolbar button aria-label, full roster in `#participants-ul`). The architectural-decision data point on captions is now in: **Option A (native scrape) is viable on Zoom** — `.lt-subtitle-wrap` + `.live-transcription-subtitle__item` are stable hooks even on a free guest meeting, with three caveats (initials-only speaker attribution, no `aria-live`, no scrolling history — all manageable). Commit covers the inventory file plus roadmap + agent-context updates.

**Exact next step (session 176): repeat the same probe-state-list workflow for Microsoft Teams.** User opens a Teams meeting URL in Chrome and walks the same lifecycle states; I send the same kind of IIFEs; selectors land in `docs/dom-inventory-teams.md`. Once Teams is logged, platform-expansion moves into adapter-implementation territory (`connectors/zoom_adapter.py`, `connectors/teams_adapter.py`). **Two paste-mangling-resistant authoring rules carried over** to apply to every Teams probe (also captured in Hard Won Knowledge): (1) use `new RegExp(string)` instead of `/regex/` literals — Chrome DevTools paste protection occasionally mangles regex/template-literal characters and the resulting syntax errors point at the wrong line; (2) match vendor aria-labels with `[aria-label*="..." i]` (contains) not exact `=`, since vendor labels are sentence-shaped, toggle-state-reflective, and sometimes typoed (Zoom's `particpants` is in production verbatim).
