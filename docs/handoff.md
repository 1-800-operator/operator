# Session 185 handoff (2026-05-05) — architecture redesign: web shrinks to marketing+signup, desktop app absorbs auth/billing/onboarding

This was a strategy + doc session. **One doc commit on local main (this session's), no production code changes.** The session-184 plan treated `1-800-operator.com` as a real web app (auth, sessions, Stripe surface, dashboard, server-side noVNC for bot account onboarding, live-meeting WebSocket fan-out). Session 185 collapsed all of that. The website is now thin and near-static: landing + email-gated signup + L0 trial-trigger endpoint + app-download CTA. **Everything authenticated lives in the desktop app talking to a unified backend over WSS** — billing (Stripe Checkout in system browser → deep-link return), dashboard, settings, meeting log, bot account onboarding (embedded Chromium first-run, replaces noVNC). **L0 free trial uses a shared pool of Operator-owned bot Google accounts** (`heyoperator001@gmail.com` …, ~2 weeks organic warmup each); user-owned bot accounts only kick in at L1+ via the in-app embedded-Chromium flow. **Backend is one unified service** for both anonymous trial-trigger endpoints and authenticated WSS — no split between "web backend" and "sidecar backend." `docs/product-strategy.md` rewritten end-to-end (TL;DR, trust ladder table, domain line, architecture diagram, "The website" + "The backend" split sections, auth/privacy two-path flow, phases A1–A10 renumbered, open questions section with new "Locked by S185 redesign" subsection). `docs/roadmap.md` and `docs/agent-context.md` got new S185 markers above the S184 markers. No new memory files needed (the redesign refines existing project memories rather than introducing new claims).
